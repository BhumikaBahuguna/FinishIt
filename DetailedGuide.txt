PART A — PROJECT MASTER EXPLANATION
--------------------------------------------------------------------

1. WHAT THE PROJECT DOES & WHY IT EXISTS
FinishIt is a productivity management application. It exists to solve the problem of overwhelming task lists and inconsistent habit tracking. Unlike a simple to-do list, it implements actual productivity frameworks:
- The Eisenhower Matrix (Urgency vs. Importance)
- Habit tracking with streak calculations
- AI-driven insights (via LLaMA 3 / Groq)
- Google Calendar Synchronization

2. ARCHITECTURE OVERVIEW
The project uses a Client-Server Architecture where the Server is a Backend-as-a-Service (BaaS) provided by Supabase.
- Frontend: React 18 built with Vite. It handles all UI, routing, and state management.
- Backend/Database: Supabase (PostgreSQL). It handles Authentication (GoTrue), Database queries (PostgREST), and data storage.
- External Services: Groq API (for the AI Assistant) and Google API (for calendar sync).

3. FULL WORKING FLOW (END-TO-END)
1. User visits the app. The React Router loads the LandingPage (`/`).
2. User clicks "Login" and goes to `/login`.
3. Authentication Flow: The `LoginPage` takes email/password and calls Supabase Auth. Supabase returns a JSON Web Token (JWT). The `AuthProvider` detects this token, saves the session, and updates the `AuthContext`.
4. Protected Routing: With an active session, the `ProtectedRoute` component allows access to `/dashboard`, `/tasks`, etc.
5. Data Fetching: When `/tasks` mounts, a `useEffect` hook fires. It calls `tasksApi.js`, which queries Supabase. Supabase returns JSON task data.
6. Rendering: React maps over the task array and renders `TaskCard` components.
7. Data Mutation: When a user adds a task, an async API call is sent to Supabase to INSERT a row. On success, the local React state is updated, immediately reflecting the new task on the screen.

PART B — FILE BY FILE CODE WALKTHROUGH
--------------------------------------------------------------------

[ROOT FILES]
1. `src/main.jsx`
- Purpose: The absolute entry point of the React app.
- Explanation: It imports React, ReactDOM, and `App.jsx`. It uses `createRoot(document.getElementById("root")).render(<App />)`. This attaches the entire React Virtual DOM to the standard HTML `<div id="root">`.

2. `src/App.jsx`
- Purpose: The Root Component.
- Explanation: Wraps the app in `AppErrorBoundary` (to catch crashes), `AppProviders` (for Auth and Context), and `AppRouter` (for navigation).
- Viva Question: "Why wrap providers around the router?" 
  Answer: "Because the Router determines which page to show based on the Auth state. If AuthProvider was inside the router, routing decisions would fail."

[ROUTING]
3. `src/router/AppRouter.jsx`
- Purpose: Maps URLs to React Components.
- Explanation: Uses React Router v6. `<Route path="/tasks" element={<TaskManagementPage />} />`. It uses wrapper components like `<ProtectedRoute>` to block unauthorized access.

[CONTEXT & PROVIDERS]
4. `src/providers/AuthProvider.jsx`
- Purpose: Manages global authentication state.
- Explanation: 
  - `useEffect`: Runs on mount to check if a session exists in localStorage/Supabase.
  - `subscribeToAuthChanges`: Listens for login/logout events from Supabase.
  - State variables: `session`, `user`, `isLoading`.
  - It provides these values to `AuthContext.Provider`.
- Viva Question: "Why use Context for Auth?"
  Answer: "Because almost every component (navbar, pages, API calls) needs to know who the user is. Passing the user prop down the entire component tree (Prop Drilling) would be a nightmare. Context makes it global."

[SERVICES (API LAYER)]
5. `src/services/aiAssistantApi.js`
- Purpose: Connects the frontend to the Groq AI API.
- Explanation: 
  - `gatherUserContext`: Fetches tasks, habits, matrix data, and formats them into a huge JSON context object.
  - `buildSystemPrompt`: Injects the context into a prompt string telling the AI how to act.
  - `sendAIMessage`: Uses the native `fetch` API to send a POST request to `https://api.groq.com/openai/v1/chat/completions` with the `VITE_GROQ_API_KEY`.
- Trap Question: "Is it secure to put the Groq API key in the frontend `.env`?"
  Answer: "Actually, no. In a true production app, exposing API keys in Vite (which ships to the browser) is insecure. A user could steal the key. For evaluation purposes, it is done this way, but the proper architecture is to have a serverless Edge Function (like Supabase Functions) hold the key and make the call."

6. `src/services/tasksApi.js`
- Purpose: Handles Supabase database interactions for tasks.
- Explanation: Uses the Supabase JS client. `supabase.from('tasks').insert([...])`. This abstracts the database logic away from the UI components.

[PAGES]
7. `src/pages/AIAssistantPage.jsx`
- Purpose: The chat interface for the AI.
- Explanation: 
  - State: `messages` (array of chat history), `input` (current typing), `isLoading`.
  - `handleSendMessage`: Appends user message to UI -> calls `sendAIMessage` -> appends AI response to UI.
- Viva Question: "How do you handle the delay while the AI is thinking?"
  Answer: "I set `isLoading(true)` before the API call, which renders a loading spinner or 'AI is typing...' indicator. In the `finally` block or after the `await`, I set `isLoading(false)`."

8. `src/pages/LoginPage.jsx`
- Purpose: User authentication.
- Explanation: Controlled form using `email` and `password` state. `onSubmit` prevents default form submission (`event.preventDefault()`) and calls `signInWithPassword`. On success, uses `navigate("/dashboard")`.

PART C — TECHNOLOGY DEEP DIVES
--------------------------------------------------------------------

1. REACT 18
- In Context: We use Functional Components and Hooks. React uses a Virtual DOM—when you update a task's state, React compares the Virtual DOM to the real DOM and only updates the specific HTML elements that changed, rather than reloading the page.

2. SUPABASE
- In Context: It provides a PostgreSQL database wrapped in an API. Instead of writing SQL in our frontend (which is impossible/insecure), Supabase provides a PostgREST API. When we call `supabase.from('tasks').select()`, it sends an HTTP GET request to Supabase, which translates it to `SELECT * FROM tasks`, and returns JSON.

3. GROQ API (LLaMA 3)
- In Context: Groq uses specialized hardware (LPUs) to run LLMs incredibly fast. We use the OpenAI-compatible endpoint. We pass "system" messages (context/instructions) and "user" messages (the prompt).

4. CSS ARCHITECTURE
- In Context: Using standard CSS classes. The project organizes CSS by logic (base, components, layouts). We use Flexbox heavily for centering and layout grids (like the Dashboard).

PART D — SYNTAX MASTERY SECTION
--------------------------------------------------------------------

1. `useEffect(() => { ... }, [])`
- Structure: Takes a callback function and a dependency array.
- Why used: For "side effects" like fetching data when a component mounts. The empty array `[]` means "run only once on mount".
- Common Viva Question: "What happens if you omit the dependency array completely?"
  Answer: "The useEffect will run after EVERY single render of the component, causing an infinite loop if state is updated inside it."

2. `const [state, setState] = useState(initialValue)`
- Structure: Array destructuring. `useState` returns an array with 2 items: the value, and the setter function.
- Why used: To hold data that changes over time and should trigger a UI update. Local variables (like `let x = 5`) do NOT trigger UI re-renders.

3. `async / await`
- Structure: `async function foo() { const res = await bar(); }`
- Why used: For handling asynchronous operations (like API calls) without using messy `.then()` promise chains. It pauses execution of the function until the promise resolves.

4. Destructuring `const { data, error } = await supabase.from...`
- Structure: Object destructuring.
- Why used: Supabase returns an object like `{ data: [...], error: null, count: 0 }`. Destructuring allows us to pull out exactly what we need cleanly.

5. Map Function `{tasks.map(task => <TaskCard key={task.id} task={task} />)}`
- Structure: Array mapping returning JSX.
- Why used: To render lists. 
- Common Viva Question: "Why is the `key` prop required?"
  Answer: "React uses the key to uniquely identify DOM elements. If a task is deleted or reordered, the key helps React efficiently update the exact DOM node instead of re-rendering the whole list."

PART E — PROJECT DEFENSE (EXPERT Q&A)
--------------------------------------------------------------------

Q1: "Why did you use React instead of vanilla HTML/JS?"
A: "Vanilla JS becomes unmanageable as UI complexity grows. React's component-based architecture allows for reusable code (like buttons and cards), and its declarative state management means I don't have to manually query the DOM (`document.getElementById`) to update data."

Q2: "Explain the data flow from database to frontend."
A: "The PostgreSQL database stores the data. Supabase exposes this via a REST API. My frontend calls this API using the Supabase JS client (`supabase.from().select()`). The returned JSON is stored in React state using `useState`. React then maps over this state to render the UI."

Q3: "How is authentication handled?"
A: "We use Supabase GoTrue Auth. When a user logs in, Supabase validates the credentials and returns a JWT (JSON Web Token). The frontend stores this token. For every subsequent database request, the Supabase client attaches this JWT in the Authorization header. Supabase reads the JWT to securely identify the user."

Q4: "What is the biggest technical challenge you faced?"
*(Good Answer)*: "Managing complex asynchronous state. For example, fetching tasks, prioritizing them into the Eisenhower matrix, and syncing them to the AI Assistant context. I solved this by strictly separating business logic into the `services/` folder, ensuring my React components remained pure and focused only on UI rendering."

TRAP QUESTION: "I see your API keys are in the frontend code. Can a malicious user access my database?"
A: "They can see the Supabase Anon Key. However, in a fully production-ready setup, we would implement Row Level Security (RLS) in PostgreSQL. With RLS, even if someone has the anon key, the database rejects any query where the JWT doesn't match the row's `user_id`. (Acknowledge that Groq API key in frontend is a vulnerability, but explain it's acceptable for a local academic project evaluation)."

PART F — SUPABASE USER CREATION DISCUSSION (CRITICAL)
--------------------------------------------------------------------

Evaluator: "What happens if a new user wants to sign up? How are users created?"

*Current Architecture Reality:*
The `LoginPage` only has a `signInWithPassword` function. There is currently no `SignUpPage`. This means the application is operating in a "Closed Beta" or "Invite Only" architecture. Users must be manually added to the Supabase Auth dashboard by the admin. 
When a user logs in for the first time, `AuthProvider.jsx` calls `ensureUserProfile`, which copies the Auth User into the `public.users` table so we can attach tasks to them.

*How to Defend This in Viva:*
"Currently, the system is designed as a closed ecosystem where administrators provision accounts via the Supabase Dashboard. This was a deliberate scope decision to prevent spam accounts during the initial release. 
If we wanted to implement open public signup, we would:
1. Create a `SignUpPage.jsx`.
2. Add a form for Email, Password, and Name.
3. Call `supabase.auth.signUp()`.
4. Handle email verification flows.
Both Supabase Auth and our `public.users` schema perfectly support this, it simply requires adding the UI component."

PART G — SYSTEM DESIGN / SCALING
--------------------------------------------------------------------

Evaluator: "What if 1 million users use this app?"

Expert Answer Blueprint:
"The current architecture is highly scalable because it's serverless, but we would need optimizations:
1. Frontend Scaling: Vercel easily scales static React assets globally via CDN.
2. Database Scaling: 1 million users means huge task tables. We would need to add pagination to our API calls (`.range(0, 19)`) so we don't fetch 10,000 tasks at once. We'd also add database indexes (which we actually already have: `idx_tasks_user_id`). Furthermore, connection pooling (PgBouncer) would be required on Supabase to handle high concurrent database connections.
3. AI Cost/Rate Limits: Hitting the Groq API for 1M users would hit rate limits. We would need to implement caching for AI responses, or process weekly summaries in batch via background jobs rather than real-time on page load."

PART I — MOCK VIVA (Comprehensive Q&A)
--------------------------------------------------------------------

[EASY]
Q: What command did you use to start the frontend server?
A: `npm run dev` (which runs the Vite development server).

Q: What is Vite?
A: A modern frontend build tool that is much faster than Create React App because it utilizes native ES modules in the browser.

Q: Where are the database tables stored?
A: They are hosted on Supabase, which is built on top of a standard PostgreSQL database.

[MEDIUM]
Q: How does the Eisenhower Matrix component determine which quadrant a task goes in?
A: In `taskPrioritization.js`, we have a function `buildEisenhowerMatrix`. It maps over tasks. If `urgency` is true and `importance` is true, it pushes to `urgent_important`.

Q: What is the purpose of the `utils` folder?
A: To hold pure JavaScript functions that don't depend on React. Things like date formatting, validation logic, or string manipulation. This keeps components clean.

Q: Explain the `public.users` table vs Supabase Auth table.
A: Supabase handles secure authentication in a hidden `auth.users` table. We create a `public.users` table to store app-specific profile data (like display names) and to act as a Foreign Key reference for our `tasks` and `habits` tables. The `AuthProvider` synchronizes the two.

[HARD]
Q: In your Supabase schema, I see `ON DELETE CASCADE` for foreign keys. Why?
A: If a user deletes their account, we want all their tasks, habits, and logs to be automatically deleted by the database engine. Without cascade, we would get foreign key constraint errors or orphaned data.

Q: Why use `useMemo` in `AuthProvider` for the context value?
A: `const value = useMemo(() => ({ session, user... }), [session])`. Every time AuthProvider re-renders, if we passed a new object literal to the Provider, it would force EVERY component consuming AuthContext to re-render, even if the user data didn't change. `useMemo` caches the object reference, preventing massive performance bottlenecks.

Q: How do you handle errors from the AI Assistant?
A: In `aiAssistantApi.js`, we wrap the `fetch` call in a `try...catch` block. If the API fails or returns a 403/500, we catch it, extract the error message, and return `{ error: err.message }`. The UI then sets an error state and displays a red banner instead of crashing the app.

Q: What happens if I modify a file in `node_modules` to fix a bug?
A: That's a terrible practice. `node_modules` is ignored by Git and recreated on every `npm install`. Any changes there will be lost. You should either write a workaround in your own code, use `patch-package`, or submit a PR to the library author.

[EXPERT / ARCHITECT LEVEL]
Q: Defend your choice of a Single Page Application (SPA) vs Server-Side Rendering (SSR) like Next.js for this project.
A: FinishIt is a highly interactive, dashboard-heavy application that sits entirely behind an authentication wall. SEO (Search Engine Optimization) is irrelevant for the internal dashboard. SPAs (React + Vite) provide a snappier, app-like experience with instant client-side routing. SSR adds backend complexity and server hosting costs that are unnecessary for a private productivity tool.

Q: If a user clicks "Complete Task" rapidly 10 times, how do you prevent 10 identical database requests?
A: By implementing "Optimistic UI Updates" with state disabling. When clicked, we immediately disable the button (using `isSubmitting` state), update the UI to show it as completed, and send ONE request to the server. We only re-enable the button once the server responds. Alternatively, we could implement a `debounce` function.

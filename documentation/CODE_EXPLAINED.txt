1. ENTRY POINT & ROOT CONFIGURATION
--------------------------------------------------------------------------------

[frontend/src/main.jsx]
Purpose: The absolute starting point of the application.
- `import ReactDOM from "react-dom/client"`: Imports the React rendering engine.
- `import "./styles/main.css"`: Imports all global CSS stylesheets.
- `ReactDOM.createRoot().render(...)`: Finds the <div id="root"> element in the 
  blank index.html file and injects the entire React application into it.
- `<React.StrictMode>`: A development tool that renders components twice to 
  catch bugs early.

[frontend/src/App.jsx]
Purpose: The root component that composes the primary application layers.
- `<AppErrorBoundary>`: If any child component crashes, this catches it and 
  displays a "Something went wrong" screen instead of a white page.
- `<AppProviders>`: Contains context providers that inject global state (like 
  the logged-in user data) into every component below it.
- `<AppRouter>`: Determines which page component to display based on the URL.

--------------------------------------------------------------------------------
2. APPLICATION INFRASTRUCTURE (src/providers/ and src/router/)
--------------------------------------------------------------------------------

[frontend/src/providers/AppProviders.jsx]
- Composes `<AuthProvider>` and `<NotificationProvider>`. Order is critical: 
  NotificationProvider needs the user's ID, so AuthProvider must wrap outside of it.

[frontend/src/router/AppRouter.jsx]
- Uses `react-router-dom` to map URLs to exact components (pages).
- Sets up `<PublicRoute>` for the Login page. 
- Sets up `<ProtectedRoute>` for Dashboard, Tasks, Habits, etc.

[frontend/src/router/ProtectedRoute.jsx] & [frontend/src/router/PublicRoute.jsx]
- `ProtectedRoute` uses `const { user } = useAuth()`. If undefined, it redirects 
  to `/login`.
- `PublicRoute` redirects authenticated users to `/dashboard`, leaving paths like 
  `/` and `/login` open only to unauthenticated visitors.

[frontend/src/layouts/MainLayout.jsx]
- The visual "shell" of the app. It renders the `<AppSidebar>` on the left side, 
  and the `<Outlet />` on the right side. This ensures the sidebar never has to 
  reload when navigating between pages.

2.1 PUBLIC PAGES & COMPONENTS (src/pages/ and src/components/)

[frontend/src/pages/LandingPage.jsx]

[frontend/src/components/ui/Section.jsx] & [frontend/src/components/navigation/Navbar.jsx]

3. SUPABASE CLIENT & AUTH API (src/services/supabase/)
--------------------------------------------------------------------------------

[frontend/src/services/supabase/client.js]
- Reads `import.meta.env.VITE_SUPABASE_URL` to securely get the database address.
- Calls `createClient(url, key)`. This object is exported and imported by EVERY 
  database file in the application to perform queries.
- `persistSession: true` ensures that closing the tab doesn't log the user out.

[frontend/src/services/supabase/authApi.js]
- Helper functions wrapper. `signInWithPassword()` runs `client.auth.signInWithPassword`.
- `subscribeToAuthChanges()` is a listener that fires a callback automatically 
  whenever a user logs in, logs out, or their token expires.

--------------------------------------------------------------------------------
4. AUTHENTICATION FEATURE (src/providers/ and src/pages/)
--------------------------------------------------------------------------------

[frontend/src/providers/AuthProvider.jsx]
- Holds state: `user`, `session`, `isLoading`.
- Inside `useEffect()`, it calls `getSession()` to check if the user is already 
  logged in when they open the app.
- It provides these variables to the `AuthContext.Provider value={value}`. Any 
  component in the app can access these variables without passing props by using 
  the `useAuth()` hook.

[frontend/src/pages/LoginPage.jsx]
- A simple form. 
- `onSubmit`: calls `signInWithPassword()`. If it fails, sets `errorMessage`. 
  If it succeeds, React Router automatically redirects to the ProtectedRoutes.

--------------------------------------------------------------------------------
5. TASKS FEATURE (src/services/ and src/pages/)
--------------------------------------------------------------------------------

[frontend/src/services/tasksApi.js]
- Functions targeting the `tasks` SQL table.
- `listTasksByUser(userId)`: Uses `client.from("tasks").select().eq("user_id", userId)`.
- `createTask()`: Inserts a new row into the table.

[frontend/src/services/taskPrioritization.js]
- The mathematical core of the Eisenhower Matrix.
- `isTaskUrgent()`: Takes the task's deadline date, subtracts Date.now(), and 
  checks if the difference is less than 48 hours (in milliseconds).
- `buildEisenhowerMatrix()`: Maps over an array of tasks and pushes them into 
  one of four arrays (DO_FIRST, SCHEDULE, DELEGATE, ELIMINATE) based on the 
  boolean values of `isUrgent` and `isImportant`.

[frontend/src/pages/TaskManagementPage.jsx]
- A monolithic page combining data listing and forms.
- Uses `useState` for the form data: title, description, deadline.
- When `loadTasks()` completes, it passes the data to `buildEisenhowerMatrix()`, 
  and then loops through the resulting four quadrants to render 4 colored boxes 
  in the UI.

6. HABITS FEATURE (src/services/ and src/pages/)
--------------------------------------------------------------------------------

[frontend/src/services/habitStreaks.js]
- `buildHabitProgressByHabit()`: This function takes a habit and an array of 
  all its `habit_logs`.
- It sorts the logs purely by date.
- It iterates backward from today. If `log.completed == true`, it `currentStreak++`. 
  The moment it hits a missing gap or a `false`, it stops.

[frontend/src/pages/HabitTrackingPage.jsx]
- Displays a list of habits.
- Each habit has a checkbox. Clicking the checkbox triggers `upsertHabitLog()`, 
  which inserts/updates the `habit_logs` SQL table for exactly "Today's Date".

--------------------------------------------------------------------------------
7. NOTIFICATIONS FEATURE (src/services/, src/providers/, and src/components/)
--------------------------------------------------------------------------------

[frontend/src/services/notificationEngine.js]
- `getNotificationsSnapshot()`: Fetches all tasks and habits from the database.
- It filters for tasks where `deadline < 4 hours from now`. It wraps these in 
  an object with `type: "upcoming_task_deadline", severity: "warning"`.
- It filters for tasks where `deadline < now`. Wraps them as `severity: "critical"`.
- It concatenates all these arrays together and sorts them so critical alerts 
  appear at the top of the list.

[frontend/src/providers/NotificationProvider.jsx]
- Uses `setInterval()` to fire the notification engine every 60,000 milliseconds (1 minute).
- Maintains a `Set` of `readNotificationIds`.
- Uses `publishNewNotifications()` which triggers browser popups for new items.

[frontend/src/components/notifications/NotificationCenter.jsx]
- The bell widget in the top layout. 
- Filters the global state `notifications.filter((n) => !n.isRead)`.
- Counts the length of that array and displays a red badge. 
- Renders a dropdown menu when `isOpen` is toggled.

--------------------------------------------------------------------------------
8. DASHBOARD OVERVIEW (src/services/ and src/pages/)
--------------------------------------------------------------------------------

[frontend/src/services/dashboardApi.js]
- Uses `Promise.all()` to fire `listHabitsByUser` and `listTasksByUser` at the 
  exact same time to speed up loading.
- Pre-calculates KPI stats (total active tasks, total overdue).

[frontend/src/pages/DashboardPage.jsx]
- Imports smaller UI components layout like `<OverviewStats />` and `<OverdueTaskView />`
  from the `components/dashboard/` directory.
- Connects the state from `dashboardApi` into the visual props of these components.

--------------------------------------------------------------------------------
9. AI ASSISTANT FEATURE (src/services/ and src/pages/)
--------------------------------------------------------------------------------

[frontend/src/services/aiAssistantApi.js]
- Acts as the bridge between the application's data and the Groq (LLaMA 3) API.
- `gatherUserContext()`: Aggregates tasks, habits, matrix data, and recent completions 
  into a comprehensive JSON object.
- `buildSystemPrompt()`: Translates the JSON context into a structured text prompt 
  that guides the AI's persona and awareness.
- `sendAIMessage()`: Performs a fetch request to the OpenAI-compatible Groq endpoint 
  and returns the parsed AI response or error.

[frontend/src/pages/AIAssistantPage.jsx]
- A dedicated chat interface with a premium, glassmorphic design.
- Uses `useState` to maintain a history of user/assistant messages.
- Implements `renderFormattedContent()` to parse markdown-like responses (bold text, 
  headers, code blocks) directly in React.
- Provides "Quick Action" buttons that trigger pre-defined, high-value prompts 
  (e.g., "Plan My Tasks").
